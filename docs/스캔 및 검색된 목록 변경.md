
## 스캔 후 검색된 목록이 표시 단순화 작업으로 변경

  1) 스캔시 파일 목록을 검색된 목록에 표시. 스캔 예외는 변경점없이 그대로 적용
  2) 검색된 목록에서 파일명마다 제품, 패치, 메뉴얼, 언어팩, 설명서로 자동분류
  3) 분류된 사항에 따라 처리
     - 제품 : ai매칭을 통해 메타데이터 생성
     - 패치 : 제품 상세페이지의 패치에 등록
     - 언어팩 : 제품 상세페이지의 언어팩에 등록
     - 메뉴얼 : 제품상세페이지의 메뉴얼 페이지에 등록
     - 업데이트 : 제품 상세페이지의 업데이트 페이지에 등록
   4) 파일명 수정, 스캔 제외 추가, 삭제는 그대로 유지
   5) 해결됨으로 표시는 사용하지 않으니 삭제

   검색된 목록 페이지에서 AI검색 및 스토어 등록, 분류 설정. 파일명 변경, 스캔 에외 추가, 삭제 아이콘 표시

   분류 설정은 위에 3) 항목을 참조

   일치/불일치 이런 항목은 판단하지 않고, 관련 소스를 삭제.


   요약하면, 파일을 스캔 후 목록을 표시 - 목록에서 분류를 통해 등록.
   파일명 규칙에 대한 것은 AI검색의 정확도를 위해 필요함. 사용자가 파일명 수정으로 통해
   규칙에 맞게 수동으로 수정 후 AI 검색을 진행하도록 해야함. 파일명이 변경될때는 실제 파일명이 변경되어야함.

   그러므로 UI도 단순화 작업이 필요하며, 분류 선택을 하면, 제품을 제외하고, 패치, 언어팩, 메뉴얼, 업데이트 분류에 따라 표시해주어야함.
   제품 상세페이지도 정보, 버전, 스크린샷, 설치방법, 패치, 언어팩, 메뉴얼, 업데이트 이렇게 표시되어야함.
   그리고 스토어의 제품상세페이지에서 버전에서 다운 가능한 버전에 목록이 나오는데, 여기서도 편집을 클릭한 후
   분류를 선택해서 해당 메뉴로 자동으로 이동되도록 하는 기능이 필요.

### 1. 현재 구조 vs 목표 구조

#### 현재 구조 (As-Is)
- **파일 스캔** → `FilenameViolation` 테이블에 위반 항목 저장
- **AI 매칭** → 신뢰도 < 90% 항목은 `UnmatchedItem` 테이블로 분리 관리
- **UI**: 파일명 위반 목록 (FilenameViolations.vue) + 불일치 검토 (UnmatchedItem) 2단계
- **제품 상세 탭**: 정보 / 버전 / 스크린샷 / 설치방법 / 자료실 (5탭, Attachment.type으로 구분)
- **Attachment.type 값**: `patch`, `manual`, `crack`, 기타 문자열

#### 목표 구조 (To-Be)
- **파일 스캔** → `ScanItem` 단일 개념으로 통합 (FilenameViolation 테이블 재활용)
- **자동 분류** → 파일명 키워드 기반으로 5가지 분류 자동 적용
- **UI**: 검색된 목록 단일 화면 (ScanList.vue), 일치/불일치 판단 제거, 해결됨 개념 제거
- **제품 상세 탭**: 정보 / 버전 / 스크린샷 / 설치방법 / 패치 / 언어팩 / 메뉴얼 / 업데이트 (8탭)
- **Attachment.type 값**: `patch`, `language_pack`, `manual`, `update`

### 2. 자동 분류 규칙 (Auto Classification)

파일명과 확장자를 분석하여 아래 우선순위 순서로 분류를 결정한다.

#### 분류 판별 우선순위

| 우선순위 | 분류 | 판별 기준 |
|---------|------|----------|
| 1 | **메뉴얼** | 확장자가 `.pdf`, `.doc`, `.docx`, `.chm`, `.txt` 이면서 파일명에 patch/update 키워드 없음 |
| 2 | **패치** | 파일명에 `patch`, `hotfix`, `fix`, `crack`, `keygen`, `serial`, `key`, `reg`, `loader` 포함 |
| 3 | **언어팩** | 파일명에 `lang`, `language`, `언어팩`, `kor_patch`, `korpatch`, `langpack` 포함 |
| 4 | **메뉴얼** | 파일명에 `manual`, `guide`, `readme`, `help`, `doc`, `tutorial`, `handbook` 포함 |
| 5 | **업데이트** | 파일명에 `update`, `upgrade`, `service_pack`, `sp[0-9]`, `cumulative` 포함 |
| 6 | **제품** | 위 규칙에 해당하지 않는 모든 파일 (기본값) |

** 폴더명에도 키워드가 있는지 반드시 체크 할 것

#### 분류 키워드 상세 (대소문자 무시)

```
제품 (product):
  - 기본값 (다른 분류에 해당하지 않는 경우)
  - 주요 확장자: .exe, .msi, .iso, .dmg, .pkg, .zip, .rar, .7z, .tar.gz

패치 (patch):
  - 키워드: patch, hotfix, fix, crack, keygen, keygenerator, serial, key,
            reg, loader, activator, unlocker, bypass
  - 확장자 무관

언어팩 (language_pack):
  - 키워드: lang, language, locale, translation, multilingual, multi_lang,
            ko_kr, en_us, ja_jp, zh_cn, zh_tw, de_de, fr_fr
  - 확장자 무관

메뉴얼 (manual):
  - 키워드: manual, guide, readme, read_me, help, doc, documentation,
            tutorial, handbook, reference, instructions
  - 확장자: .pdf, .doc, .docx, .chm, .txt (키워드 없어도 이 확장자이면 메뉴얼로 분류)

업데이트 (update):
  - 키워드: update, upgrade, service_pack, sp1, sp2, sp3, cumulative,
            rollup, release, build
  - 확장자 무관
```

#### 폴더명 활용
- 파일명이 `setup.exe`, `install.exe` 등 의미 없는 경우 부모 폴더명을 우선 검색 컨텍스트로 사용 (기존 로직 유지)

### 3. DB 스키마 변경

#### 3-1. FilenameViolation 테이블 → ScanItem으로 재활용

기존 `filename_violations` 테이블에 컬럼 추가 (Alembic 마이그레이션):

```sql
ALTER TABLE filename_violations
  ADD COLUMN classification VARCHAR(20) DEFAULT 'product',
  -- 값: product / patch / language_pack / manual / update
  ADD COLUMN classification_auto BOOLEAN DEFAULT TRUE;
  -- TRUE: 자동 분류, FALSE: 사용자가 수동으로 변경한 분류
```

기존 컬럼 중 제거 또는 미사용 처리:
- `is_resolved`: 소프트 삭제 방식으로 유지하되 UI에서 노출 제거 (기존 데이터 호환성)
- `violation_type`, `violation_details`, `suggestion`: 기존 필드 유지 (마이그레이션 최소화)

#### 3-2. UnmatchedItem 테이블 제거

- `unmatched_items` 테이블과 관련 API/프론트엔드 코드를 제거
- 불일치 판단 로직 (`confidence_score` < 90% → UnmatchedItem 생성) 제거
- 스캔 시 모든 파일은 `filename_violations`(= ScanItem)에 단순 등록

#### 3-3. Attachment.type 값 정규화

기존 `patch`, `manual`, `crack`, 기타 → 아래 4가지로 표준화:

| 타입 | 설명 | 표시 탭 |
|------|------|---------|
| `patch` | 패치, 크랙, 키젠 등 | 패치 탭 |
| `language_pack` | 언어팩, 번역팩 | 언어팩 탭 |
| `manual` | 메뉴얼, 설명서, 가이드 | 메뉴얼 탭 |
| `update` | 업데이트, 서비스팩 | 업데이트 탭 |

기존 데이터 마이그레이션: `crack` → `patch`, 미분류 → `patch` (기본값)

#### 4-2. filename_violations.py API 변경

제거할 기능:
- `POST /{id}/resolve` (해결됨 표시) → 제거
- `GET /stats` 중 `resolved_count`, `mismatch_count` 필드 → 제거
- UnmatchedItem 연동 코드 전체 제거

추가/변경할 기능:
```
GET  /api/scan-items/              - 스캔 목록 조회
     ?classification=product|patch|language_pack|manual|update (필터)
     ?search=키워드 (파일명 검색)
     ?page=1&size=50 (페이지네이션)
     응답: [{id, folder_path, file_name, classification, classification_auto, created_at}]

PATCH /api/scan-items/{id}/classify
     body: {classification: "patch"}  - 분류 수동 변경
     → classification_auto = FALSE로 업데이트

POST /api/scan-items/{id}/register
     - classification에 따라 자동 등록 처리
     - 제품: AI 매칭 후 Product 생성 (기존 create-product 로직 재사용)
     - 패치/언어팩/메뉴얼/업데이트: 연결할 Product 선택 후 Attachment 생성
     body: {product_id: 123}  (제품 이외 분류일 때 필수)

POST /api/scan-items/{id}/rename   - 파일명 변경 (기존 로직 유지, 실제 파일 rename)
POST /api/scan-items/{id}/exclude  - 스캔 제외 추가 (기존 로직 유지)
DELETE /api/scan-items/{id}        - 스캔 항목 삭제 (기존 로직 유지)
POST /api/scan-items/batch-delete  - 일괄 삭제

GET /api/scan-items/stats
    응답: {total, by_classification: {product, patch, language_pack, manual, update}}
```

#### 4-4. 버전 → Attachment 이동 API

```
     처리: Version 레코드 삭제 → Attachment 레코드 생성 (file_path, file_name 유지)
```

### 8. 분류 변경 UX 흐름 정리

```
[검색된 목록] 화면
     ↓ 분류 배지 드롭다운 클릭
  분류 선택 (제품/패치/언어팩/메뉴얼/업데이트)
     ↓
  [제품 선택] → AI검색 → [스토어 등록]  ← 제품 분류인 경우
  [제품 검색 다이얼로그] → [등록]        ← 패치/언어팩/메뉴얼/업데이트인 경우

     ↓ 편집 드롭다운 → "패치로 분류" 선택
  자동으로 현재 제품에 Attachment 생성
  (다른 제품이면 제품 선택 다이얼로그 표시)
     ↓
  "패치 탭으로 이동할까요?" 확인 후 탭 전환
```
